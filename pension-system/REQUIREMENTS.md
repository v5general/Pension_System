## 1. 背景（前因后果）

### 1.1 社会背景

随着我国人口老龄化程度不断加深，城镇及乡镇地区老年人口数量持续增长。据国家统计局数据显示，我国60岁及以上人口占比已超过18%，且呈现持续上升趋势。这一人口结构变化对社会养老服务体系建设提出了严峻挑战。

### 1.2 现状问题

当前城镇/乡镇地区老年人养老情况的管理工作面临以下突出问题：

#### 1.2.1 数据管理层面
- **统计方式落后**：传统统计方式以纸质记录、分散Excel表格为主，缺乏系统化管理
- **数据整合效率低**：不同部门、不同时期的数据难以快速整合分析
- **数据安全风险高**：纸质文件易丢失、损坏，缺乏备份机制
- **汇总展示不直观**：数据统计耗时耗力，难以快速生成可视化报告

#### 1.2.2 隐私保护层面
- **公开性与隐私性难以平衡**：汇总统计数据需要向社会公开，但个人具体敏感信息需要严格保护
- **权限管理缺失**：缺乏完善的用户权限体系，无法精确控制数据访问范围

#### 1.2.3 民意收集层面
- **诉求收集渠道缺失**：老年人的养老诉求、生活困难等问题缺乏统一的收集渠道
- **信息反馈滞后**：政府/社区难以及时、精准掌握辖区内养老需求
- **政策制定缺乏数据支撑**：养老服务优化、政策调整缺乏可靠的数据依据

### 1.3 建设契机

为响应国家"智慧养老"政策号召，提升基层养老治理能力，亟需建设一套数字化、智能化的养老数据管理系统。通过信息化手段整合养老数据、畅通民意渠道、支撑科学决策，切实提升养老服务的精准性和有效性。

## 2. 项目愿景（目标、解决的问题）

### 2.1 核心目标

搭建一套**轻量化、易操作、安全可控**的"智慧养老数据库"Web系统，实现城镇/乡镇老年人养老数据的统一管理、民意调查的高效收集、养老问题的定向反馈，为政府决策和社区服务提供数据支撑。

### 2.2 解决的问题

#### 2.2.1 数据管理问题
| 问题 | 解决方案 |
|------|----------|
| 养老数据分散存储，难以统一管理 | 建立集中式数据库，实现数据统一存储 |
| 统计效率低，汇总耗时费力 | 自动生成汇总统计，支持多维度查询 |
| 数据展示不直观，难以快速理解 | 提供可视化图表展示，支持数据导出 |
| 纸质记录易丢失、损坏 | 数据库持久化存储，支持定期备份 |

#### 2.2.2 隐私保护问题
| 问题 | 解决方案 |
|------|----------|
| 个人隐私数据与公开数据难以区分 | 实现数据分级管理，汇总数据公开、个人数据仅管理员可见 |
| 缺乏有效的访问控制机制 | 建立角色权限体系，精确控制数据访问范围 |
| 敏感信息泄露风险 | 身份证号、联系方式等敏感字段仅管理员可查看 |

#### 2.2.3 民意收集问题
| 问题 | 解决方案 |
|------|----------|
| 缺乏统一的民意收集渠道 | 提供在线民意调查功能，随时收集反馈 |
| 老年人诉求难以传达 | 建立问题收集模块，支持文字描述反馈 |
| 需求响应不及时 | 管理员可实时查看、分类处理问题 |

#### 2.2.4 决策支撑问题
| 问题 | 解决方案 |
|------|----------|
| 政策制定缺乏数据依据 | 提供多维度统计分析，为决策提供支撑 |
| 养老服务优化方向不明确 | 基于民意调查结果，精准识别服务需求 |
| 资源分配缺乏量化标准 | 通过数据分析，优化养老资源配置 |

### 3.2 一般用户

#### 3.2.1 角色定位
一般用户指需要查看养老统计数据、参与民意调查、提交养老问题的普通用户，包括社区工作者、调查员、老年人家属等。

#### 3.2.2 核心权限

| 权限类别 | 具体权限 | 权限说明 |
|----------|----------|----------|
| **账户管理** | 注册账号 | 使用用户名、密码、手机号注册系统账户 |
| | 登录系统 | 输入用户名、密码登录系统 |
| | 修改密码 | 可修改个人登录密码 |
| | 修改用户名 | 可修改自己账号显示的用户名 |
| | 注销账号 | 可注销自己的账号（需二次确认，不可恢复） |
| **数据查看** | 查看养老数据汇总 | 可查看各地区养老数据的统计汇总结果（如养老金均值、疾病占比等） |
| | 筛选查看 | 支持按地区、年龄段等条件筛选查看汇总数据 |
| | 导出汇总数据 | 可将汇总统计结果导出为Excel/CSV格式 |
| **民意调查** | 填写民意调查 | 可填写并提交民意调查表单 |
| | 查看调查结果 | 可查看民意调查的汇总统计结果 |
| **问题收集** | 提交问题 | 可填写并提交养老问题、困难或建议 |
| | 查看个人提交记录 | 可查看自己提交的问题及处理状态 |
| **页面导航** | 页面跳转 | 可在系统各页面之间正常跳转 |

#### 3.2.3 权限限制
- **禁止**查看具体个人的养老数据（姓名、身份证号、联系方式等）
- **禁止**查看其他用户提交的民意调查原始数据
- **禁止**查看其他用户提交的问题收集内容
- **禁止**进行数据的录入、修改、删除操作
- **禁止**访问管理员专属页面

### 3.3 管理员

#### 3.3.1 角色定位
管理员指拥有系统完整管理权限的用户，负责养老数据的录入维护、民意调查的统计分析、问题收集的处理反馈等工作。

#### 3.3.2 核心权限

| 权限类别 | 具体权限 | 权限说明 |
|----------|----------|----------|
| **一般用户权限** | 继承一般用户所有权限 | 管理员拥有一般用户的所有权限 |
| **养老数据管理** | 查看具体数据 | 可查看每个老年人的详细养老数据 |
| | 录入具体数据 | 可录入新的老年人养老数据 |
| | 修改具体数据 | 可修改已录入的养老数据 |
| | 删除具体数据 | 可删除错误或过期的养老数据 |
| | 批量导入数据 | 支持通过Excel批量导入养老数据 |
| | 数据导出 | 可导出养老数据为Excel/CSV格式 |
| **民意调查管理** | 查看原始数据 | 可查看所有用户提交的民意调查原始数据 |
| | 按条件筛选 | 支持按提交时间、问题类型等条件筛选查看 |
| | 数据导出 | 可导出民意调查原始数据 |
| **问题收集管理** | 查看所有问题 | 可查看所有用户提交的问题收集内容 |
| | 更新处理状态 | 可将问题状态更新为"已查看"、"已解决"等 |
| | 添加处理备注 | 可对问题的处理情况添加备注说明 |
| | 问题分类统计 | 可按问题类型进行统计分析 |
| **用户账号管理** | 查看用户列表 | 可查看系统所有注册用户 |
| | 查看用户状态 | 可查看所有用户的在线状态（在线/离线/注销） |
| | 修改用户角色 | 可将一般用户提升为管理员或降级为一般用户 |
| | 重置用户密码 | 可帮助用户重置忘记的密码 |
| | 禁用/启用账号 | 可禁用违规账号或重新启用 |
| | 注销用户账号 | 可强制注销一般用户的账号（不可恢复） |
| **账户管理** | 修改自己的用户名 | 可修改自己账号显示的用户名 |
| | 注销自己的账号 | 可注销自己的账号（需二次确认，不可恢复） |

## 4. 主要的业务流程

### 4.1 业务流程概述

系统包含四大核心业务流程：
1. **养老数据管理流程**：管理员录入维护养老数据，系统自动生成汇总统计
2. **民意调查流程**：用户填写调查问卷，系统汇总统计调查结果
3. **问题收集流程**：用户提交问题建议，管理员查看处理反馈
4. **页面跳转流程**：用户登录后在各功能模块间导航

## 5. 数据

### 5.1 数据概述

系统数据分为四大类别：

#### 5.2.2 字段详细说明

**个人信息类字段**
- **姓名**：必填，老年人真实姓名
- **身份证号**：必填，唯一标识，18位身份证号码，用于数据去重和身份验证
- **年龄**：必填，整数，60周岁及以上
- **性别**：必填，枚举值（男/女/其他）

**住址信息类字段**
- **住址**：必填，详细地址，需包含省/市/区/乡镇/社区信息
- **区域类型**：必填，区分城镇/乡镇，用于分类统计

**家庭状况类字段**
- **家庭人口数**：可选，填写则需大于等于1
- **有无子女**：必填，有/无/不详
- **子女数量**：当"有无子女"选择"有"时建议填写
- **婚姻状况**：必填，未婚/已婚/丧偶/离异

**经济状况类字段**
- **有无工作**：必填，有/无/已退休
- **有无退休金**：必填，有/无
- **每月养老金金额**：必填，无则填0，单位：元
- **有无养老保险**：必填，有/无
- **保险类型**：有养老保险时填写，城乡居民/职工养老/其他
- **子女赡养费金额**：必填，无则填0
- **赡养费周期**：默认按月，可选按年

**健康医疗类字段**
- **有无疾病**：必填，有/无
- **疾病类型**：有疾病时建议填写，可填写多种疾病（如高血压、糖尿病等）

**联系方式类字段**
- **联系方式**：可选，手机号或家庭电话，仅管理员可见

**系统管理类字段**
- **数据录入时间**：自动生成，记录数据首次录入时间
- **数据更新时间**：自动更新，记录数据最后一次修改时间
- **录入人ID**：关联用户表，记录数据录入操作人

#### 5.2.3 数据隐私分级

#### 6.2.1 SQLite数据库连接接口

**接口说明**：系统通过GORM与SQLite数据库进行交互。

**技术实现**：
- **数据库**：SQLite 3.x
- **连接库**：`gorm.io/driver/sqlite`
- **ORM框架**：GORM v1.25+
- **连接方式**：文件型数据库，无需独立服务

**配置示例**（database/db.go）：
```go
package database

import (
    "gorm.io/driver/sqlite"
    "gorm.io/gorm"
    "gorm.io/gorm/logger"
)

// InitDB 初始化数据库连接
func InitDB(dbPath string) (*gorm.DB, error) {
    db, err := gorm.Open(sqlite.Open(dbPath), &gorm.Config{
        Logger: logger.Default.LogMode(logger.Info),
    })
    if err != nil {
        return nil, err
    }
    return db, nil
}
```

**管理工具**：
- 数据库管理：Navicat Premium / DB Browser for SQLite
- 数据库备份：直接复制.db文件

**操作流程**：

1. 用户点击"注销账号"按钮
2. 系统显示警告提示："注销后您的所有数据将被永久删除，无法恢复！"
3. 用户输入"确认注销"文本进行二次确认
4. 系统执行注销操作：
   - 删除用户账号记录
   - 清除该用户的投票数据
   - 清除该用户提交的问题记录
   - 保留该用户提交的民意调查数据（用于统计）
5. 注销成功后跳转到登录页

**安全措施**：
- 需要输入当前密码验证身份
- 需要手动输入"确认注销"文本
- 显示明确的不可恢复警告
- 管理员账号不能被注销（需降级为一般用户后才能注销）

#### 9.1.2 技术选型理由

**后端选择Go + Wails的原因**：
1. 性能优越：Go语言编译后的执行效率高，内存占用低
2. 跨平台编译：一次编译，多平台运行，打包发布简单
3. 并发处理：Go原生支持高并发，适合数据处理密集型应用
4. 部署便捷：Wails打包后为单一可执行文件，无需额外依赖
5. 类型安全：静态类型语言，编译期捕获错误，运行更稳定
6. 现代化：Wails使用现代前端技术栈，开发体验好

**前端选择Vue3 + Vite的原因**：
1. 开发体验好：Vite冷启动快，热更新迅速
2. 组件化开发：Vue3 Composition API代码组织更灵活
3. 生态成熟：Element Plus提供完整的企业级UI组件
4. 响应式设计：天然支持响应式数据绑定
5. 学习成本低：中文文档完善，社区活跃

**数据库选择SQLite的原因**：
1. 零配置：无需安装数据库服务，开箱即用
2. 轻量级：单文件存储，便于备份和迁移
3. 性能足够：对于中小型应用性能完全满足需求
4. 跨平台：数据文件可在不同平台间通用
5. 易维护：使用Navicat等工具可视化管理，操作简单

package database

import (
    "gorm.io/driver/sqlite"
    "gorm.io/gorm"
    "gorm.io/gorm/logger"
    "pension-system/backend/models"
)

// DB 数据库连接包装
type DB struct {
    *gorm.DB
}

// InitDB 初始化数据库连接
func InitDB(dbPath string) (*DB, error) {
    db, err := gorm.Open(sqlite.Open(dbPath), &gorm.Config{
        Logger: logger.Default.LogMode(logger.Info),
    })
    if err != nil {
        return nil, err
    }

    return &DB{DB: db}, nil
}

// AutoMigrate 自动迁移数据库表
func AutoMigrate(db *DB) error {
    return db.AutoMigrate(
        &models.User{},
        &models.Elderly{},
        &models.Survey{},
        &models.SurveyAnswer{},
        &models.Issue{},
        &models.Statistics{},
    )
}

// Close 关闭数据库连接
func (db *DB) Close() error {
    sqlDB, err := db.DB.DB()
    if err != nil {
        return err
    }
    return sqlDB.Close()
}

// database/seed.go
package database

import (
    "pension-system/backend/models"
    "golang.org/x/crypto/bcrypt"
)

// SeedData 初始化数据
func SeedData(db *database.DB) error {
    // 检查是否已存在管理员
    var count int64
    db.Model(&models.User{}).Where("username = ?", "admin").Count(&count)
    if count > 0 {
        return nil
    }

    // 创建默认管理员账户
    hashedPassword, _ := bcrypt.GenerateFromPassword([]byte("admin123"), bcrypt.DefaultCost)
    admin := models.User{
        Username: "admin",
        Password: string(hashedPassword),
        Role:     "admin",
        RealName: "系统管理员",
    }
    return db.Create(&admin).Error
}

### 10.7 用户手册

#### 10.7.1 一般用户手册

**智慧养老数据库系统 - 用户手册**

**1. 系统简介**

智慧养老数据库系统用于管理老年人养老数据、收集民意调查和养老问题反馈。

**2. 用户注册与登录**

*2.1 注册账户*
1. 访问系统首页
2. 点击"注册"按钮
3. 填写用户名、密码、手机号
4. 点击"提交"完成注册

*2.2 登录系统*
1. 访问系统首页
2. 输入用户名和密码
3. 点击"登录"按钮

**3. 查看养老数据汇总**

1. 登录后，点击顶部导航"养老数据汇总"
2. 可查看各地区老年人统计数据
3. 支持按地区、年龄、性别筛选查看
4. 可导出汇总数据为Excel

**4. 填写民意调查**

1. 点击"民意调查"菜单
2. 填写每月开销构成（五项总和需为100%）
3. 选择最急需解决的问题
4. 选择是否愿意进养老院
5. 填写政策建议（可选）
6. 点击"提交"完成调查

**5. 提交养老问题**

1. 点击"问题收集"菜单
2. 选择问题类型
3. 填写问题标题和详情
4. 点击"提交"完成提交
5. 可在"我的问题"中查看处理状态

#### 10.7.2 管理员操作指南

**管理员操作指南**

**1. 管理员登录**

使用管理员账户登录系统，可访问所有功能模块。

**2. 养老数据管理**

*2.1 录入数据*
1. 点击"具体数据管理"
2. 点击"新增数据"按钮
3. 填写养老数据表单
4. 点击"提交"完成录入

*2.2 查询数据*
1. 在数据管理页面设置查询条件
2. 点击"查询"按钮
3. 系统显示符合条件的数据列表

*2.3 修改数据*
1. 在数据列表中点击"编辑"
2. 修改需要变更的字段
3. 点击"保存"完成修改

*2.4 删除数据*
1. 在数据列表中点击"删除"
2. 确认删除操作

**3. 查看原始调查数据**

1. 点击"民意调查"→"原始数据"
2. 可查看所有用户提交的调查数据
3. 支持按时间和问题类型筛选

**4. 处理问题收集**

1. 点击"问题收集"→"问题管理"
2. 查看所有用户提交的问题
3. 点击"处理"按钮
4. 更新处理状态和添加处理备注
5. 点击"保存"完成处理

**5. 数据导出**

1. 在对应数据页面点击"导出"按钮
2. 选择导出格式（Excel/CSV）
3. 保存文件到本地
